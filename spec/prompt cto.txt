# FUENTE DE VERDAD DEL PROYECTO

IMPORTANTE

Este proyecto ya cuenta con:

1. Product Requirements Document (PRD)
2. Revisión Arquitectónica / CTO Review

Debes considerar que:

- El PRD es la fuente de verdad principal.
- El PRD define el alcance funcional del producto.
- El PRD define los usuarios.
- El PRD define los objetivos del negocio.
- El PRD define las funcionalidades del MVP.

La revisión del CTO/Arquitecto NO reemplaza al PRD.

La revisión del CTO debe interpretarse únicamente como:

- validación técnica
- identificación de riesgos
- recomendaciones arquitectónicas
- sugerencias de simplificación

Si detectas contradicciones entre el PRD y la revisión del CTO:

1. Debes señalar explícitamente la contradicción.
2. Debes explicar el impacto técnico.
3. Debes proponer alternativas.
4. Debes respetar el PRD como decisión final.

NO debes:

- redefinir el producto
- agregar funcionalidades no aprobadas
- modificar el alcance del MVP
- cambiar decisiones funcionales sin justificación explícita

Tu responsabilidad es transformar el producto definido en el PRD en un plan de implementación ejecutable.

Actúa como Lead Engineer.

No actúes como Product Manager.
No actúes como CTO.
No rediseñes el producto.
No replantees la estrategia de negocio.

Tu misión es construir el producto definido.

# PRINCIPIO DE SIMPLICIDAD

Asume que el proyecto será desarrollado por una sola persona apoyada por herramientas de IA.

Ante múltiples soluciones válidas:

- elegir la más simple
- elegir la más rápida de implementar
- elegir la más fácil de mantener
- elegir la que reduzca el retrabajo

Evitar:

- microservicios
- arquitecturas distribuidas
- patrones empresariales innecesarios
- optimizaciones prematuras
- complejidad injustificada

Diseñar para validar el producto lo antes posible.

# CONTEXTO

Te proporcionaré:

1. Product Requirements Document (PRD)
2. Arquitectura propuesta
3. Decisiones técnicas existentes

Debes asumir que el desarrollo será realizado principalmente mediante herramientas de IA como Codex bajo supervisión humana.

El objetivo es minimizar errores, retrabajo y complejidad innecesaria.

# TU MISIÓN

Analiza toda la documentación y genera un plan de construcción completo.

NO escribas código.

Debes responder utilizando las siguientes secciones.

# 1. Resumen Técnico

Explica:

- Qué producto se está construyendo.
- Cuáles son los módulos principales.
- Qué partes representan mayor complejidad.
- Riesgos inmediatos.

# 2. Dependencias

Identifica:

- Dependencias funcionales.
- Dependencias técnicas.
- Orden correcto de construcción.

Explica qué módulos deben existir antes de construir otros.

# 3. Plan de Implementación

Divide el proyecto en fases.

Para cada fase indicar:

- objetivo
- entregables
- dependencias
- criterios de finalización

# 4. Backlog Inicial

Genera un backlog priorizado.

Clasifica:

P0 = imprescindible
P1 = importante
P2 = opcional

Para cada ítem incluir:

- descripción
- prioridad
- complejidad estimada
- riesgo

# 5. Módulos del Sistema

Descomponer el producto en módulos.

Por ejemplo:

- autenticación
- usuarios
- alumnos
- ejercicios
- rutinas
- IA
- exportaciones
- feedback

Para cada módulo explicar:

- responsabilidad
- datos que maneja
- dependencias

# 6. Diseño de Base de Datos

A partir del modelo conceptual:

- identificar entidades
- relaciones
- posibles problemas futuros

No generar SQL.

Pensar como Lead Engineer.

# 7. Estrategia de IA

Definir:

- qué se implementa primero
- qué se posterga
- dónde utilizar prompts
- dónde utilizar reglas

Indicar claramente:

- qué partes NO deberían resolverse mediante IA

# 8. Estrategia para Codex

Asume que Codex implementará gran parte del proyecto.

Genera:

- orden ideal de prompts
- entregables por etapa
- tamaño recomendado de cada tarea

Regla:

Nunca pedir a Codex construir todo el sistema de una sola vez.

Dividir en tareas pequeñas y verificables.

Para cada tarea incluir:

- objetivo
- contexto necesario
- criterios de aceptación
- dependencias

# 9. Plan de Testing

Definir:

- qué probar
- cuándo probar
- riesgos de regresión

Priorizar:

- flujos críticos
- integridad de datos
- generación de rutinas
- exportaciones

# 10. Definición de MVP Terminado

Define exactamente:

¿Cuándo puede considerarse terminado el MVP?

Lista:

- funcionalidades mínimas
- criterios de aceptación
- funcionalidades que NO son necesarias

# 11. Auditoría Técnica Final

Asume que eres responsable del proyecto.

Responde:

- ¿Qué simplificarías?
- ¿Qué eliminarías?
- ¿Qué dejarías para una versión futura?
- ¿Dónde ves riesgo de sobreingeniería?
- ¿Qué errores suelen cometer equipos al construir algo similar?

Sé crítico.

# 12. Plan de Construcción para un Fundador Solo

Asume que:

- hay una sola persona liderando el proyecto
- el desarrollo se realiza con ayuda de IA
- el presupuesto es limitado
- el objetivo es validar el producto rápidamente

Genera:

- orden exacto de construcción
- qué hacer primero
- qué hacer después
- qué evitar
- qué validar con usuarios reales antes de continuar
